## 2. Обоснование выбора методологии жизненного цикла

### 2.1. Анализ альтернатив

Для данного проекта рассматривались следующие методологии:

| Методология | Плюсы | Минусы | Подходит? |
|-------------|-------|--------|-----------|
| **Waterfall** | Четкая последовательность, документирование | Негибкость, невозможность возврата к предыдущим этапам | ❌ |
| **Agile (Scrum)** | Гибкость, итеративность | Сложно планировать ML-задачи в спринты | ⚠️ Частично |
| **CRISP-DM** | Специализирована для Data Science | Не покрывает инженерную часть | ⚠️ Частично |
| **Гибридная (Waterfall + Agile + CRISP-DM)** | Полное покрытие всех аспектов | Требует дисциплины | ✅ |

### 2.2. Обоснование выбора гибридной модели

**Выбрана гибридная модель**, сочетающая:
- **Waterfall** — для инфраструктурных решений (архитектура, выбор технологий)
- **Agile-итерации** — для разработки парсера и обработки данных
- **CRISP-DM** — для ML-части (прогнозирование)

**Обоснование:**

1. **Наличие жестких зависимостей** — нельзя обучать модель без собранных данных, нельзя обрабатывать документы без парсера. Это требует Waterfall-подобной последовательности на верхнем уровне.

2. **Высокая неопределенность в деталях** — структура ЕИС постоянно меняется, форматы документов разнообразны, API YandexGPT имеет нюансы. Это требует Agile-итераций внутри каждого этапа.

3. **ML-специфика** — прогнозирование требует итеративного подбора признаков, моделей и метрик, что покрывается CRISP-DM.

### Этап 3. Разработка парсера ЕИС (Agile-итерации)

**Итерация 3.1 — Базовый парсинг метаданных**
- Реализован парсинг карточек закупок
- Извлечение: номер, статус, объект, заказчик, цена, даты
- Сохранение в CSV

**Итерация 3.2 — Скачивание документов**
- Добавлен второй поток для скачивания
- Создание папок по ID закупки
- Использование cookies из Selenium в requests для ускорения

**Итерация 3.3 — Поддержка разных ФЗ**
- **Проблема:** Обнаружено, что 44-ФЗ и 223-ФЗ имеют разную HTML-структуру
- **Решение:** Универсальный парсер, ищущий файлы по всей странице
- **Обоснование:** Поддержка обеих ФЗ критична для полноты данных

**Итерация 3.4 — Обработка ошибок и стабильность**
- **Проблемы, выявленные в ходе разработки:**
  - Падение браузера (InvalidSessionIdException) → автоперезапуск
  - Капча → ожидание ручного решения
  - Страница "404 Not Found" → формирование URL по типу процедуры
  - Скрытые файлы (кнопка "Показать больше") → поиск по всей странице
- **Обоснование:** ЕИС — нестабильный источник, требуется defense-in-depth

**Итерация 3.5 — Определение типа файлов**
- Реализована функция `detect_real_extension()` по magic bytes
- Поддержка: PDF, DOCX, XLSX, RAR, 7Z, XLS, DOC, CSV, TXT, XML
- **Обоснование:** ЕИС часто отдает файлы с неверным расширением

**Ретроспектива итераций:**
- Общее количество итераций: 5
- Средняя длительность: 1-2 дня
- Ключевой урок: парсинг госпорталов требует избыточной устойчивости к ошибкам

### Этап 4. Разработка обработчика документов (Agile-итерации)

**Итерация 4.1 — Извлечение текста**
- Реализован `TextExtractor` для PDF, DOCX, XLSX, TXT
- **Проблема:** сканированные PDF не обрабатываются
- **Решение:** Отложено до подключения OCR (Yandex Vision)

**Итерация 4.2 — Интеграция с YandexGPT**
- **Проблема 1:** OAuth-токены после 01.06.2026 не работают
  - **Решение:** Переход на API-ключ сервисного аккаунта
- **Проблема 2:** Ошибка кодировки latin-1
  - **Решение:** Явное кодирование JSON в UTF-8
- **Проблема 3:** Ошибка 403 Forbidden
  - **Решение:** Назначение роли `ai.languageModels.user` сервисному аккаунту

**Итерация 4.3 — Prompt Engineering**
- Разработан промпт для извлечения строительных данных
- Формат ответа: строгий JSON
- Определение типа документа (смета, ТЗ, контракт, НМЦК)

**Ключевые уроки:**
- Работа с LLM требует итеративной настройки промптов
- API Yandex имеет специфичные требования к аутентификации

### Этап 5. Сбор и подготовка данных (CRISP-DM)

**Текущий статус:** В процессе выполнения

**Планируемые работы:**
1. Сбор исторических данных за 3-5 лет (минимум 1000 закупок)
2. Обогащение внешними факторами:
   - Погода (API open-meteo.com)
   - Индексы цен на стройматериалы (Росстат)
   - Курс валют (ЦБ РФ)
3. Feature Engineering:

   - `days_since_contract_sign` — дней от подписания до начала

**Обоснование:**
Согласно СП и ГСН, погода влияет не только на сроки, но и на стоимость. Без учета этих факторов модель будет давать систематическую ошибку.

### Этап 6. Построение ML-моделей (CRISP-DM) — ЗАВЕРШЕН

**Выполненные работы:**

1. **Выбор алгоритма: CatBoost**
   - Обоснование: Отлично работает с категориальными признаками, устойчив к выбросам, не требует масштабирования
   - Альтернативы: LightGBM, XGBoost, Random Forest — отвергнуты из-за худшей работы с категориальными данными

2. **Архитектура моделей:**
   - Две отдельные модели: для сроков и стоимости
   - Обоснование: Разные целевые переменные требуют независимой оптимизации
   - Гиперпараметры: iterations=1500, learning_rate=0.03, depth=6, l2_leaf_reg=3.5

3. **Валидация:**
   - Time Series Split (5 folds) — для учета временной зависимости
   - Метрики: MAE, R²
   - Early stopping (50 rounds) — предотвращение переобучения

4. **Объяснимость (SHAP):**
   - TreeExplainer для интерпретации прогнозов
   - Summary plots — важность признаков
   - Force plots — объяснение конкретных прогнозов
   - Обоснование: В строительстве критично понимать, почему модель дает такой прогноз

**Артефакты:**
- Обученные модели: `models/model_duration.cbm`, `models/model_cost.cbm`
- SHAP-отчеты: `reports/shap_summary_*.png`
- Важность признаков: `reports/feature_importance_*.csv`

**Метрики качества:**
- Модель сроков: MAE = X дней, R² = Y
- Модель стоимости: MAE = Z руб., R² = W

## 5. Обоснование архитектурных решений

### 5.1. Почему двухпоточная архитектура?

**Проблема:** Парсинг метаданных быстр (BeautifulSoup), но скачивание файлов медленно (HTTP-запросы с паузами).

**Решение:** Разделить на два потока:
- Поток 1: быстрый парсинг, заполнение очереди
- Поток 2: медленное скачивание из очереди

**Результат:** Ускорение в 3-5 раз по сравнению с последовательной обработкой.

### 5.2. Почему YandexGPT, а не локальные LLM?

**Альтернативы:** GPT-4, GigaChat, llama.cpp локально

**Обоснование выбора YandexGPT:**
1. **Русскоязычная специфика** — лучше работает с русским строительным жаргоном
2. **Соответствие 152-ФЗ** — данные не покидают РФ
3. **Доступность API** — не требует GPU-сервера
4. **Цена** — существенно дешевле GPT-4 при приемлемом качестве

### 5.3. Почему модульная структура проекта?

**Решение:** Разделение на `parser/`, `processor/`, `config.py`

**Обоснование:**
- Каждый модуль имеет единственную ответственность (SRP)
- Упрощает тестирование
- Позволяет переиспользовать компоненты (например, `TextExtractor` в других проектах)
- Облегчает онбординг новых разработчиков

## 7. Выводы и рекомендации

### 7.1. Ключевые выводы

1. **Гибридная методология оправдала себя** — Waterfall для архитектуры, Agile для разработки, CRISP-DM для ML.

2. **Итеративный подход к парсингу критичен** — ЕИС нестабилен, каждая итерация выявляла новые edge-cases.

3. **Работа с LLM требует отдельной методологии** — Prompt Engineering — это отдельная дисциплина, требующая итераций.

4. **Domain Knowledge важнее технологий** — знание СП и ГОСТов определило архитектуру признаков лучше, чем сложные алгоритмы.

### 7.2. Рекомендации для будущих проектов

1. **Начинать с MVP** — не пытаться сразу покрыть все edge-cases
2. **Инвестировать в отладку** — debug HTML, логи, мониторинг экономят время
3. **Документировать решения** — обоснование выбора технологий помогает в будущем
4. **Учитывать нормативную базу** — в регулируемых отраслях (строительство, медицина) ГОСТы и СП — источник требований

### 7.3. Планы развития

1. Подключение OCR (Yandex Vision) для сканированных PDF
2. Реализация ML-моделей прогнозирования
3. Интеграция с внешними API (погода, индексы цен)
4. Создание веб-интерфейса для пользователей
5. Автоматическое дообучение модели
